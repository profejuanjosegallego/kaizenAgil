Primero la BD

La tabla usuarios

Toda entidad nace de una tabla. Columnas en español y snake_case, la llave primaria como UUID y el correo único.

El modelo

La clase Usuario

La misma tabla, ahora como clase Java. Cada anotación tiene un propósito. Usamos los campos en snake_case para que coincidan 1:1 con las columnas (convención de nuestra clase).

Usuario.java 
@Entity
@Table(name = "usuarios")
public class Usuario {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @Column(nullable = false, length = 120)
    private String nombre;

    @Column(nullable = false, unique = true, length = 160)
    private String correo;

    @Column(name = "contraseña_hash", nullable = false)
    private String contraseña_hash;

    @Column(nullable = false, length = 20)
    private String rol;               // "docente" | "estudiante"

    @Column(nullable = false)
    private Boolean activo = true;

    @Column(name = "color_avatar", length = 9)
    private String color_avatar;

    @Column(name = "fecha_registro")
    private LocalDateTime fecha_registro;

    // Constructores, getters y setters...
}

Anotaciones de mapeo

@Entity marca la clase como tabla.
@Table(name) nombre real de la tabla.
@Id llave primaria.
@GeneratedValue(UUID) genera el id solo.
@Column ajusta nombre, tipo, nullable, unique, length.

¿Por qué UUID y no un número?

El UUID es único globalmente, no revela cuántos registros hay y evita choques al integrar módulos entre equipos.

El corazón del modelo

Relaciones entre entidades

Un Usuario publica muchas Prendas. Esa es una relación uno a muchos. La llave foránea (id_vendedor) vive en la tabla del lado “muchos”.

@Entity
@Table(name = "prendas")
public class Prenda {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @Column(nullable = false)
    private String titulo;

    private Double precio;

    // Lado DUEÑO: aquí vive la llave foránea.
    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "id_vendedor")
    private Usuario vendedor;

Usuario.java · lado inverso (mappedBy) 
// Dentro de la clase Usuario:

// Lado INVERSO: NO tiene FK, solo "apunta de vuelta".
@OneToMany(
    mappedBy = "vendedor",        // nombre del campo en Prenda
    cascade = CascadeType.ALL,    // guarda/borra en cascada
    orphanRemoval = true          // borra prendas huérfanas
)
private List<Prenda> prendas_publicadas = new ArrayList<>();

@JoinColumn

Va en el lado dueño (@ManyToOne). Define el nombre de la columna FK: id_vendedor.

mappedBy

Va en el lado inverso (@OneToMany). Dice qué campo del otro lado es el dueño. No crea columna.

fetch = LAZY

Carga la relación solo cuando se usa. Evita traer datos de más. Recomendado en @ManyToOne.

Letra pequeña

Anatomía de una relación

Cada atributo dentro de @ManyToOne / @OneToMany tiene un trabajo concreto. Aquí está la relación completa, atributo por atributo.

la relación, anotada 
// LADO DUEÑO (Prenda): tiene la llave foránea
@ManyToOne(fetch = FetchType.LAZY,    // cuándo se carga el vendedor
           optional = false)          // la FK es obligatoria (NOT NULL)
@JoinColumn(name = "id_vendedor")     // nombre de la columna FK
private Usuario vendedor;

// LADO INVERSO (Usuario): NO tiene columna FK
@OneToMany(mappedBy = "vendedor",         // campo dueño en Prenda
           cascade = CascadeType.ALL,     // propaga guardar/borrar a las prendas
           orphanRemoval = true)          // borra la prenda si sale de la lista
private List<Prenda> prendas_publicadas = new ArrayList<>();

fetch — ¿cuándo se cargan los datos?

LAZY Perezoso: la relación se trae solo cuando la usas. Ahorra memoria y consultas. Recomendado.

EAGER Ansioso: se trae siempre, junto con la entidad. Cómodo, pero puede traer datos de más y causar el problema “N+1”.

Relaciónfetch por defecto
@ManyToOneEAGER
@OneToOneEAGER
@OneToManyLAZY
@ManyToManyLAZY

cascade — ¿qué se propaga al hijo?

Si haces algo con el padre (Usuario), cascade repite esa operación en los hijos (sus prendas). Sin cascade, tendrías que guardar/borrar cada prenda a mano.

CascadeTypePropaga…
PERSISTguardar (insertar)
MERGEactualizar
REMOVEborrar
REFRESHrecargar desde la BD
DETACHdesvincular del contexto
ALLtodas las anteriores

mappedBy

Va en el lado inverso. Dice qué campo del otro lado es el dueño ("vendedor"). No crea columna; solo indica “yo no mando”.

optional = false

La relación es obligatoria: la FK queda NOT NULL. Cada Prenda debe tener vendedor. Por defecto es true (opcional).

orphanRemoval = true

Si sacas una prenda de la lista del usuario, queda huérfana y se borra de la BD. Ideal cuando el hijo no existe sin su padre.

cascade REMOVE vs orphanRemoval — no confundir

cascade = REMOVE borra los hijos cuando borras al padre completo. orphanRemoval = true borra un hijo concreto cuando lo quitas de la colección (aunque el padre siga vivo). Para entidades fuertemente dependientes, se suelen usar juntas.

El truco

¿Quién es la dueña de la relación?

Regla de oro

La dueña es la tabla que guarda la llave foránea (la del @JoinColumn). El otro lado es el inverso y usa mappedBy.

En una relación 1 — N ni siquiera tienes que elegir: la FK solo cabe en el lado “muchos”. Cada Prenda guarda una id_vendedor… pero en usuarios no podrías poner una sola columna que apunte a muchas prendas. Por eso el dueño es siempre el lado @ManyToOne.

Usuario · inversa
  • id : UUID
  • @OneToMany(mappedBy)
  • ✕ sin columna FK
la FK manda →
Prenda · DUEÑA
  • id : UUID
  • @ManyToOne · @JoinColumn
  • ✓ columna id_vendedor

Según el tipo de relación

Relación¿Quién es la dueña?¿Se elige?
@ManyToOne / @OneToMany (1—N)El lado @ManyToOne (el “muchos”, donde está la FK)No: forzado
@OneToOne (1—1)El lado donde pongas @JoinColumnSí: tú decides
@ManyToMany (N—N)El lado con @JoinTableSí (ustedes la evitan)

3 preguntas para decidir

1. ¿En qué tabla cabe la columna FK? → ahí está la dueña.
2. ¿Es 1—N? → la dueña es el lado N, sin discusión.
3. ¿Es 1—1? → la dueña es la entidad dependiente (la que “no existe sin la otra”).

Por qué importa al guardar 
// CORRECTO: seteas el lado DUEÑO → la FK se guarda
prenda.setVendedor(usuario);
prendaRepository.save(prenda);

// INSUFICIENTE: solo el inverso → la relación NO se persiste
usuario.getPrendas_publicadas().add(prenda); // por sí solo no basta

En una frase

“La dueña tiene la llave”. Donde vive @JoinColumn está la dueña; el otro lado solo mappedBy. Y al guardar, mueve siempre la dueña.

Buena práctica

@ManyToMany o tabla intermedia?

La recomendación

Sí: casi siempre evita @ManyToMany y modela el muchos‑a‑muchos como una entidad intermedia (tabla puente). Conviertes el N–N en dos 1–N que sí controlas.

1 — N

N — 1

@ManyToMany · evítalo 
// Funciona, PERO la tabla "favoritos" es implícita:
// no puedes agregarle columnas (fecha, estado...).
@ManyToMany
@JoinTable(
    name = "favoritos",
    joinColumns = @JoinColumn(name = "id_usuario"),
    inverseJoinColumns = @JoinColumn(name = "id_prenda"))
private List<Prenda> favoritas = new ArrayList<>();

Sin control de la tabla de unión: el día que necesites fecha_marcado, toca rehacerlo.

Favorito.java · recomendado 
@Entity
@Table(name = "favoritos")
public class Favorito {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "id_usuario")
    private Usuario usuario;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "id_prenda")
    private Prenda prenda;

    // El dato extra que justifica la tabla puente:
    @Column(name = "fecha_marcado")
    private LocalDateTime fecha_marcado;
}

Controlas la tabla, agregas columnas y consultas con claridad. Usuario y Prenda tendrían @OneToMany(mappedBy=...) hacia Favorito.

Por qué la entidad intermedia gana

  • Puedes añadir columnas propias de la relación (fecha, cantidad, estado, precio).
  • Controlas fetch, cascade y las consultas.
  • Es a prueba de futuro: si la relación “crece”, ya está modelada.

¿Cuándo SÍ se vale @ManyToMany?

Solo para asociaciones puras, sin ningún dato extra y que no vayan a crecer (ej. Prenda ↔ Etiqueta). Aun así, muchos equipos prefieren la entidad intermedia por precaución.

El error más común

Recursión infinita: qué es y por qué pasa

Antes de arreglarlo, hay que entenderlo. La recursión infinita aparece cuando dos objetos se apuntan el uno al otro y, al convertirlos a JSON, el proceso no termina nunca.

¿Qué es?

Un bucle sin fin: A contiene a B, B contiene a A, A contiene a B… El mismo par de objetos se repite indefinidamente.

¿Por qué se da?

Porque la relación es bidireccional: Usuario conoce sus prendas y cada Prenda conoce a su vendedor (¡el mismo usuario!).

¿Por qué controlarla?

Si no, la API se cae con StackOverflowError o devuelve un JSON gigante y duplicado.

Paso a paso: cómo nace el bucle

1. El controlador devuelve un Usuario y Jackson (Java → JSON) recorre sus atributos.

2. Llega a prendas_publicadas → serializa cada Prenda.

3. Cada Prenda tiene vendedor → serializa otra vez al Usuario.

4. Ese usuario tiene prendas_publicadas → vuelve al paso 2… para siempre.

5. La pila de llamadas se llena → StackOverflowError.

Por qué NO se puede dejar así

  • Tumba el endpoint: StackOverflowError → error 500, la API queda inservible.
  • JSON enorme y duplicado: si no desborda, repite los mismos datos y consume memoria, red y tiempo.
  • Filtra datos sensibles: al serializar la entidad completa podrías exponer campos como contraseña_hash.
  • Acopla tu API: la respuesta queda atada a la estructura interna de tus entidades.

La solución

Cómo cortar el ciclo

Ya sabes por qué pasa. La idea para arreglarlo siempre es la misma: romper la cadena en uno de los dos lados para que la serialización termine.

Usuario prendas_publicadas vendedor ∞ se corta aquí

El par estándar: @JsonManagedReference + @JsonBackReference

Usuario · lado que SÍ se serializa 
@OneToMany(mappedBy = "vendedor")
@JsonManagedReference          // lado "gestionado": SÍ aparece en el JSON
private List<Prenda> prendas_publicadas;
Prenda · lado que NO se serializa 
@ManyToOne
@JoinColumn(name = "id_vendedor")
@JsonBackReference             // lado "de vuelta": se OMITE → corta el ciclo
private Usuario vendedor;

¿Qué se usa más hoy? — respuesta directa

Sí, @JsonManagedReference/@JsonBackReference y @JoinColumn son lo clásico, pero hay más opciones. De menos a más recomendado:

  • @JsonManagedReference + @JsonBackReference: par estándar para relaciones bidireccionales. El lado "back" no se serializa.
  • @JsonIgnore: lo más rápido. Oculta el campo en el JSON (siempre).
  • @JsonIgnoreProperties("..."): ignora solo ciertas propiedades del otro lado. Más fino.
  • @JsonIdentityInfo: en vez de omitir, referencia por id; útil para grafos complejos.
  • DTO (lo más recomendado hoy): no serializas la entidad; devuelves un objeto plano. Cero riesgo de ciclos y controlas exactamente qué expones.
Alternativas rápidas 
// Opción simple: ocultar el lado "de vuelta"
@ManyToOne @JoinColumn(name = "id_vendedor")
@JsonIgnore
private Usuario vendedor;

// Opción por id (rompe ciclos sin ocultar nada):
@JsonIdentityInfo(
    generator = ObjectIdGenerators.PropertyGenerator.class,
    property = "id")
@Entity
public class Usuario { /* ... */ }

Acceso a datos

El repositorio JPA

Con solo extender JpaRepository ya tienes el CRUD. Y con métodos derivados (nombrados de cierta forma) Spring Data escribe el SQL por ti.

UsuarioRepository.java 
@Repository
public interface UsuarioRepository extends JpaRepository<Usuario, UUID> {

    // Métodos derivados: el nombre del método define la consulta.
    Optional<Usuario> findByCorreo(String correo);

    List<Usuario>    findByActivoTrue();
    List<Usuario>    findByRol(String rol);
    List<Usuario>    findByNombreContainingIgnoreCase(String nombre);

Tip con campos snake_case

Los métodos derivados funcionan perfecto con campos de una palabra (correo, rol, activo). Para campos de varias palabras como fecha_registro, lo más claro es usar @Query (siguiente slide) y evitar ambigüedades de nombres.

Consultas personalizadas

@Query: JPQL vs SQL nativo

Cuando el nombre del método no alcanza, escribes la consulta con @Query. Hay dos sabores: JPQL (habla de clases y atributos) y SQL nativo (habla de tablas y columnas).

JPQL · orientado a objetos 
// Usa la CLASE (Usuario) y sus atributos (u.correo)
@Query("SELECT u FROM Usuario u " +
       "WHERE LOWER(u.correo) = LOWER(:correo)")
Optional<Usuario> buscarPorCorreo(
       @Param("correo") String correo);

JPQL

Trabaja con entidades, no con tablas. Es portable entre bases de datos. :correo es un parámetro nombrado con @Param.

SQL nativo · sobre la tabla real 
// Usa la TABLA (usuarios) y columnas reales
@Query(value = "SELECT * FROM usuarios " +
               "WHERE activo = TRUE AND rol = :rol",
       nativeQuery = true)
List<Usuario> activosPorRol(
       @Param("rol") String rol);

SQL nativo

Es SQL puro (nativeQuery = true). Útil para consultas muy específicas del motor, pero menos portable.

¿Quién llama al repositorio?

El Servicio (@Service) usa el repositorio, aplica las reglas y devuelve un DTO al Controlador (@RestController). El repositorio nunca habla directo con el cliente.

Regla de oro

La FK va con @JoinColumn en el lado @ManyToOne; el otro lado usa mappedBy. Para exponer datos por la API, devuelve DTOs, no entidades. Y si serializas entidades bidireccionales, corta el ciclo con @JsonManagedReference/@JsonBackReference.
